The DCAT-Applicationprofile for Dataportals in Switzerland (DCAT-AP-CH) is an application profile of the DCAT vocabulary ([[VOCAB-DCAT]]). It serves to describe Data that have been registered on Swiss Dataportals. It targets managers on opendata portals in Switzerland and the publishers of these data. It is in use since 2016 in the Federal dataportals. It has to be considered that the dataportals don't offer the data themselves, but instead offer metadata on the data.

The catalog of a dataportal has 3 hierarchical levels of entries: There is a description of the catalog itself, metadata entries on the data collection that are gathered in the portal and metadata on the various distributions of these data. The DCAT applications profile for Dataportals in Switzerland aims at being compatible with the Application profile for dataportals in Europe DCAT-AP.

Introduction

The DCAT-Applicationprofile for Dataportals in Switzerland (DCAT-AP-CH) is an application profile of the DCAT vocabulary ([[VOCAB-DCAT]]).It serves to describe Data that have been registered on Swiss Dataportals.

DCAT is a semantic definition to describe Data by the means of an RDF-vocabulary. It allows for a decentralized approach regarding the publication of catalogs and data. It has been developed for open geovernment data, in oder to facilitate the data and metadata reuse and the interoperability between various data catalogs.

DCAT-AP-CH serves to the describe data collections and their distributions. These descriptions have the purpose to make the data discoverable and not to describe the data itself.

DCAT-AP-CH targets managers on opendata portals in Switzerland and the publishers of these data. Since 2016 it has been in use on the portal opendata.swiss to describe the data registered by the portal

A data catalog conforms to DCAT-AP-CH if:

  • It contains all mandatory classes specified in the specification
  • An RDF description of the catalog itself is available
  • All classes and properties defined in the catalog are used in a way consistent with the semantics declared in this specification.
DCAT-AP-CH compliant catalogs MAY include additional non-DCAT-AP-CH metadata fields and additional RDF data in the catalog's RDF description.

Multilanguages, URIs and Content Negociation

Switzerland is a multilingual country. Therefore supporting multiple languages is very important in DCAT-AP-CH. See the below on how multiple languages are handled in the Swiss Application profile:

  • Multiligual Literals: Properties of Range rdfs:Literal support multiple languages out of the box and the Swiss Profile makes use of this. It also adds some additional restraints on what languages are expected.
  • Content Negotian: Properties of Range rdfs:Resource are supposed to be URIs. It is good practice to use URIs that are language independent ao that content negotiation can be used at these URI to provide the desired language to users that follow them.

Best practice for Resource URIs They should be language and format neutral: http://my-dataset-name-language-and-format-neutral Then content negotiation can be applied to get the desired language and format.

Applcation profile overview

DCAT-AP-CH scope

DCAT-AP-CH uses only some of the classes of [[DCAT-AP]]. It also uses Classes from [[FOAF]] for modeling of the Agent class.

  • dcat:Catalog is a container of datasets in DCAT-AP-CH.
  • dcat:Dataset is a container of distributions.
  • dcat:Distribution represents an accessible form of a dataset such as a downloadable file.
  • foaf:Agent is an Organization that is authorized to publish data.
  • There are other classes in [[DCAT-AP]] that have not yet become part of DCAT-AP-CH

Vocabulary specification

The tables in the following sections include these columns:

  • Column Obl. ("Obligation"), denotes whether the class / property is mandatory ("M"), conditional (C) or optional (O) - where conditional means "mandatory if given conditions are met".
  • Column MC specifies the maximum cardinality of the corresponding property (the minimum cardinality is always 1 for all mandatory properties, and 0 for the other ones). Properties having as range a plain literal, can be specified multiple times, even though their max cardinality is 1, provided that each instance uses a different language tag. These case is marked by using an asterisk after the max cardinality value (e.g., "1*").
  • Column Conf. ("Conformance"), denotes whether the class / property definition is conformant ("Y") or not conformant (N) to [[DCAT-AP]]

Primary Classes

Class: Catalog

Label Catalogue
Obligation Mandatory
RDF Class dcat:Catalog
Definition A Datacatalog is a collection of metadata entries to dataset that are hosted on a dataportal.
Usage The catalog has a hierarchical structure and contains references to all the metadata for the data that can be accessed via the portal (The entries have the class dcat:Dataset). The datasets themselves refer to all Distributions of the actual data (class dcat:Distribution). In DCAT-AP there is an additional class dcat:CatalogRecord, which contains Metadata on the Catalog. This class has not been used so far in the implementation of DCAT-AP-CH on opendata.swiss.

        Label
        Dataset

          offered in different ways: as file in a text/csv format or as API Service. Each Distribution
          SHOULD be described by a dcat:Distribution. In general each dataset will have at least

          and it must be considered that [[VOCAB-DCAT]] goes more in the direction of
          allowing datasets to have parts (that could even be catalogs) then in the
          direction of datasets having distribution as parts.

          Label
          identifier
        

        

          Obligation
          Mandatory
        

        

          Max card.
          1
        

        

          RDF Property
          dct:identifier
          
        

        

          Range
          rdfs:Literal
          
        

        

          Definition
          Unique identifier of the dataset.
            The identifier maybe used as part of the URI of the dataset,
            but having it represented explic-itly is still useful.
          
        

        

          Usage
          
It is necessary to uniquely mark each Dataset in order to be able to link it
            to similar datasets. The identifier of a dataset should therefore have
            the following structure:

            
ID_Datenset@ID_Datapublisher

            
Example: 
              124325@swisstopo
            

            
The identifier should be unique within Switzerland

            
The dataaset can then be linked to other dataset by the element
              rdfs:seeAlso by adding the identifier to this property
              on other datasets.

            
Using rdfs:seeAlso for linking to other datasets, makes it necessary that
              a dataportal implementing this automatically takes care of consistency
              if datasets get changed or removed:

            
    
              
  • If the identifier of a dataset is changed: all seeAlsos of
                other datasets on the portal should be changed accordingly
              
    • 
              
  • If a dataset is deleted: its identifier should be removed
                of the seeAslso of other datasets.
              
    • 
            

          		
        

        

          Conformance
          The Usage of this property is not [[DCAT-AP]] comformant. Also rdfs:seeAlso is not DCAT-AP
            conformant. Instead dct:relation should be used for that purpose: In DCAT-AP the property is called
            "related resource" dct:relation. It has range rdfs:Resource
            and the description says: "This property refers to a related
            resource." The DCAT-AP links this property to a
            
              discussion about grouping datasets on Joinup

Conformance to DCAT-AP


  

    
This document is the preparation for version 2 of DCAT-AP-CH. The goal is to reach conformity with the DCAT-AP
      while remaining in line with the Swiss demands of OGD and NaDB.

  


  

    
Criteria


    

      That DCAT-AP-CH as a DCAT profile conforms to DCAT-AP means that it
      generally follows the spirit of DCAT and DCAT-AP. It may add additional constraints
      on classes and properties, but it should not enforce whatever DCAT-AP enforces as a minimum.
      Where DCAT-AP demands controlled vocabulary it is strongly recommended to use this
      vocabulary. It can define custom properties to cover special Swiss needs, but it
      should not invent constructs and properties that strech DCAT into unintended directions


  

  


    
Overview


    
When studying DCAT-AP and DCAT-AP-CH in close detail many problems surface where
      DCAT-AP-CH does not conform. Some are are easy to fix and not a big deal, why others are more severe
      and hint on a differnt understanding of the overall structure of the Metadata-Model that needs to be addressed,
      before DCAT-AP-CH can profit from new developments in DCAT


    
In this section you will find a short overview of the problems that will then be analysed in more
      detail in the sections below.


    
    
      
  • Different understanding of the role of Datasets and Distributions: The most severe problem
        is grounded in a different understanding of the role of dcat:Dataset and dcat:Distribution:
        while in DCAT all Distributions are supposed to have the same content and different content such as
        series of content always needs datasets that might be connected, in DCAT-AP-CH distributions are supposed
        to model series of similar data. This general understanding has consequences:
        in DCAT there is no coverage for Distributions. Coverage can only happen on the dataset level.
        In DCAT-AP-CH on the other hand the attribute dct:coverage was introduced for exactly that
        purpose: for expressing what part of the dataset a resource covers. I would suggest to
        leave that property behind and follow DCAT in this. Otherwise in the long run Distribution will become
        sort of Mini Dataset and that will cause problems with other properties too.
      
    • 
      
  • Vocabulary usage: In some properties DCAT-AP demands the usage
        of EU vocabulary, where in DCAT-AP-CH the ranges have been defined using DCAT vocabulary. This
        is true for the language property () as well as
        for the frequency (()). This problems are easy to fix:
        as well in the definition as in the implemetation on opendata.swiss.
      
    • 
      
  • Inconsistencies in the implementation on opendata.swiss:
        Some of these inconsistencies might be misundertstandings. Therfore examples have been
        added to this specification, to clarify how the properties should be used.
        Some of this incoistencies may just resolve by themselves such as the fact that DCAT-AP-Ch
        specifies Literals of tape date for issuance and modification date, while DCAT-AP also
        allows datetime for these field and they have been implemented as dattime.
        Also the homepage has been misunderstood, as it is implemented as a Literal on opendata.swiss
        where it should be a Resource in DCAT-AP-CH as in DCAT-AP.
      
    • 
      
  • Modelling relationships between datasets
    • 
      
  • Licence and rights
    • 
      
  • Format and media type
    • 
      
  • The modeling of Relationship of resources has flaws
        rdfs:seeAlso is modeled in way that goes against the spirit of Linked Data.

Concept of a Dataset issue



      
Some problems that we currently face, for example the need of the attribute dct:coverage
        for dcat:Distribution come from the fact, that we we misinterpret what is meant by a dataset: in [[VOCAB-DCAT]]
        there is a note
        dcat:Distribution

      that states that the Distributions of a dataset should contain the same content, but in different
      format. If there is a series of different content it should rather be modeled as a datasets that relate
      to each other. 

      
It seems painful to change this defintion what a dataset is now, but in the long run we will be better
        of if we adapt to [[VOCAB-DCAT]]'s generel concepts. Only this will enable us to profit in the best
        way from its further development.
