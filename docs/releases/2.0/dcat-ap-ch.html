The DCAT Application Profile for data portals in Switzerland DCAT-AP CH is an application profile of the DCAT vocabulary [[VOCAB-DCAT]]. It serves to describe data that has been registered on Swiss data portals. Its target group are managers of open data portals in Switzerland and the providers of these data. DCAT-AP CH has been used by the Open Government Data Portal of the Swiss Federal Statistical Office since 2016.

It is important to be aware that data portals don't provide the data themselves, only metadata on the data.

The catalog of a data portal has three hierarchical levels: a description of the catalog, metadata entries on the collections of data, and metadata on the various physical distributions of these data.

The DCAT Application Profile for data portals in Switzerland is mostly compatible with the Application Profile for data portals in Europe [[DCAT-AP]].

About

The DCAT Application Profile for data portals in Switzerland (DCAT-AP CH) is an application profile of the DCAT vocabulary ([[VOCAB-DCAT]]). It serves to describe data that have been registered on Swiss data portals.

DCAT [[VOCAB-DCAT]] is a semantic definition to describe data by the means of an RDF vocabulary. It allows for a decentralized approach regarding the publication of data by enforcing interoperabilty via the use of a common language to describe the data. It has been developed for open government data, in order to facilitate the reuse of this data.

Since DCAT is only a language, it can be applied in many contexts. An application profile applies DCAT to one specific context. In the case of DCAT-AP CH the context is that of publishing open data on dataportals in Switzerland.

This context alone would allow for a very flexible usage of DCAT in DCAT-AP CH, but Switzerland is also part of Europe and therefore the interoperability with the dataportal of the European Union has been a topic from the start. Europe has it's own application profile [[DCAT-AP]], that has established a role of a de facto standard within Europe. Swiss data is also harvested by and represented on the European data portal. The EU puts a lot of power behind further developing their application profile DCAT-AP. Therefore staying in line with the DCAT-AP is not only a goal in itself, but means that DCAT-AP CH will stay embeded in a bigger community of stakeholders.

Metadata structure

DCAT vocabulary generelly consists of classes and properties. These are used to describe metadata, that is data about data.

  • Classes are things on the internet: Not all of them have URIs, but it is reccommended to provided an URI for them. They are complex things like a person, an organization, a data collection, a website, a downloadable data file.
  • Classes have properties: The properties are the attributes describing these things. Some properties occur on more than one class, a title for example is a common attribute. Other properties are very specialized such as a file format describing a datafile.
  • Properties can be simple or complex: Some properties are classes. For example an organization can have a website. Or a dataset can have a data publisher. In general a class can be recognized by its spelling: A property name starts with a lower case letter such as dcat:dataset (), while a class starts with a capital letter such as dcat:Dataset ().
  • The main classes of DCAT-AP CH are the classes used to describe a data catalog: They are called primary classes and they consist of , and . Secondary classes are classes that are used in properties, for exapmple the data publisher of a dataset as foaf:Organization or the landing page of a dataset as dcat:LandingPage.
  • In DCAT-AP CH the primary classes form a hierarchy: A catalog () is a collection of data at a data portal. A catalog bundles datasets: a dataset () is a collection of similar data. And a distribution is a description of a source of data that is at the bottom of this hierarchy the source where the data can directly accessed.
  • DCAT is disolving these hierarchies While the distribution as the actual data source remains a leaf of the of the data catalog, the overall structure of the data catalog, can be a lot more complex in DCAT and DCAT-AP. Interoperability

DCAT-AP CH aims at making data discoverable on the national and international level by providing the nessary metadata in a form that can be understood by both humans and machines.

A precondition this for interoperability is that the DCAT language is used correctly. Therefore this specification aims at enabling datapublishers to specify their data catalog in a way that conforms to both DCAT-AP CH and DCAT-AP.

The first part of the conformance statement (above this note) is generated by [[RESPEC]] as it is considered an integral part of any specification. The second part (below this note) has been added by the editor and follows similar conformance notes in [[VOCAB-DCAT]] and [[DCAT-AP]]. opendata.swiss currently does not conform as an application. For a discussion on this see .

A data catalog conforms to DCAT-AP CH if:

  • An RDF description of the catalog is available
  • All classes and properties defined in DCAT-AP CH are used in a way consistent with the semantics declared in this specification.
  • For any additional classes and properties the conformance rules of DCAT ( DCAT Conformance Criteria) and [[DCAT-AP]] (Section 6.1 "Provider requirements") apply.

An application (data portal) conforms to DCAT-AP CH if:

  • It is able to process RDF catalogs that conform to DCAT-AP CH.
  • Processing means that it must accept incoming data and transparently provide this data to applications and services.
  • Processing does not prescribe whether and how the metadata is stored internally, displayed or made searchable.
  • For any additional classes and properties, especially those coming from [[DCAT-AP]] the conformance rules of [[VOCAB-DCAT]] and [[DCAT-AP]] (Section 6.2 "Receiver requirements") apply: the application must be able to import and export classes and properties that are well formed according to the semantics of in DCAT-AP correctly and pass them through as they are when harvested by the Europen dataportal.

Multiple languages, URIs and content negotiation

This section has been added in order to give more guidance on the use of URIs and how they can help with translations.

Switzerland is a multilingual country. Therefore, supporting multiple languages is very important in DCAT-AP CH. Multiple languages are handled as follows.

  • Multilingual literals: Properties of Range rdfs:Literal support multiple languages and the Swiss Profile makes use of this. It also adds some additional constraints on what languages are expected.
  • Content negotiation: Properties of Range rdfs:Resource should be URIs. It is good practice to use URIs that are language independent, so that content negotiation can be used at these URIs to provide the desired language to users that follow them.

Best practice for Resource URIs They should be language and format neutral: http://my-dataset-name-language-and-format-neutral. Content negotiation can then be applied to get the desired language and format.

Vocabulary specification

This vocabulary specification has been added in order to provide a legend for the class property overview tables that have been added for each class.

The tables in the following sections include these columns:

  • Column Obl. ("Obligation"), denotes whether the class / property is mandatory ("M"), conditional (C) or optional (O). Conditional means that the property becomes mandatory under certain conditions.
  • Column MC specifies the maximum cardinality of the corresponding property (the minimum cardinality is always 1 for all mandatory properties, and 0 for the other ones). Properties having as range a plain literal can be specified multiple times, even though their max cardinality is 1, provided that each instance uses a different language tag. This case is marked by using an asterisk after the max cardinality value (e.g., "1*").
  • Column Conf. ("Conformance"), denotes whether the class / property definition is conformant ("Y") or not conformant (N) to [[DCAT-AP]].

Conformance to DCAT-AP


  

    
This document is the preparation for version 2 of DCAT-AP CH. The goal is to reach conformity with DCAT-AP
      while remaining in line with the Swiss demands of [[OGD]] and [[NaDB]]. 

    
Therefore this
    section explains the conformance criteria and points out conformance violations, so that they can
      be further discussed and solved by the community around DCAT-AP CH.

  


  

    
Conformance Criteria


    
DCAT-AP CH and DCAT-AP are both application profiles for DCAT.

    
The [[DCAT-PROFILE-GUIDANCE]] states that application profiles may form hierarchies.

    
For interoperabilty reasons DCAT-AP CH aims to be a subprofile of DCAT-AP. That way
    the data using DCAT-AP CH can be harvested by the European dataportal without loosing
    important information on its metadata.


    
The following diagramm captures the relationship between DCAT, DCAT-AP and DCAT-AP CH:

    

      classDiagram

      DCAT -->  DCAT_AP: is_profile_of
      DCAT_AP --> DCAT_AP_CH: is_subprofile_of
      DCAT --> DCAT_AP_CH: is_profile_of
      class DCAT {
        - defines classes
        - defines properties
        - base vocabulary
      }
      class  DCAT_AP {
        - mandatory/recommended classes
        - mandatory/recommended properties
        - mandatory/recommended vocabulary
      }
      class DCAT_AP_CH {
        - can restrict more than DCAT_AP
        - cannot restrict less than DCAT_AP
      }
    


    
DCAT-AP CH must conform to both DCAT and DCAT-AP.
      This conformance of DCAT-AP CH means in detail:

    
    
      
  • Conformance to DCAT: All classes and properties MUST be used in a way that they were meant for by the DCAT vocabulary:
      For example: DCAT vocabulary states that the distributions of a dataset should only differ in resolution,
      language and format but not in the content that they cover. Therefore an attrbitute such as
        dct:coverage on dcat:Distribution hurts that principle and is discussed
        in this section as being not conformant to DCAT. See
         for the details.
      
    • 
      
  • Conformance to DCAT:
        DCAT-AP CH MAY introduce Swiss specific properties and vocabulary, such as the cantons of Switzerland, that
        don't exist in the DCAT vocabulary.
        It should however not introduce its own (custom) properties that are already expressed in DCAT with different vocabularies.
        In those cases the DCAT vocabulary has to be used.
        For example the Swiss rdfs:seeAlso is used
        to express relationships between datasets. But DCAT already has vocabulary to express
        those relationships. Therefore these Swiss constructs are not conforming to DCAT.
        These relationships can not be interpreted on an international or European level.
        See .
      
    • 
      
  • Conformance to DCAT-AP:
        DCAT-AP CH can restrict more than DCAT-AP regarding mandatory fields and mandatory vocabulary, but it is not allowed
        to restrict less. For example currently DCAT-AP CH uses DCAT vocabulary for many properties, such as dct:language,
        dct:accrualPeriodicty. This usage conforms DCAT, but not DCAT-AP. Therefore these vocabulary usages are also
        considered to be a conformance issue. See  and
        .
