This has been translated from the section "Zusammenfassung" of [[DCAT-AP-CH-1]]

The DCAT Application Profile for data portals in Switzerland (DCAT-AP CH) is an application profile of the DCAT vocabulary ([[VOCAB-DCAT]]). It serves to describe data that have been registered on Swiss data portals. It is directed to managers on open data portals in Switzerland and the publishers of these data, and has been used by the Federal data portals since 2016.

It is important to be aware that data portals don't provide the data themselves, only metadata on the data. The catalog of a data portal has 3 hierarchical levels of entries: a description of the catalog itself, metadata entries on the collections of data (datasets) that are gathered in the portal, and metadata on the various distributions of these data. The DCAT Application Profile for data portals in Switzerland aims at being compatible with the Application Profile for data portals in Europe, DCAT-AP.

Introduction

This has been translated from the section "1.2 Anwendungsgebiet" of [[DCAT-AP-CH-1]]

The DCAT Application Profile for data portals in Switzerland (DCAT-AP CH) is an application profile of the DCAT vocabulary ([[VOCAB-DCAT]]). It serves to describe data that have been registered on Swiss data portals.

DCAT is a semantic definition to describe Data by the means of an RDF vocabulary. It allows for a decentralized approach regarding the publication of catalogs and data. It has been developed for open government data, in order to facilitate reuse of data and metadata and interoperability between various data catalogs.

DCAT-AP CH serves to describe data collections and their distributions. These descriptions have the purpose of making the data discoverable, not of describing the data themselves.

DCAT-AP CH is directed to managers on open data portals in Switzerland and the publishers of these data. Since 2016, it has been in use on the portal opendata.swiss to describe the data registered by the portal.

Multiple languages, URIs and content negotiation

This section has been added in order to give more guidance on the use of URIs and how they can help with translations.

Switzerland is a multilingual country. Therefore, supporting multiple languages is very important in DCAT-AP CH. Multiple languages are handled as follows.

  • Multilingual literals: Properties of Range rdfs:Literal support multiple languages and the Swiss Profile makes use of this. It also adds some additional constraints on what languages are expected.
  • Content negotiation: Properties of Range rdfs:Resource should be URIs. It is good practice to use URIs that are language independent, so that content negotiation can be used at these URIs to provide the desired language to users that follow them.

Best practice for Resource URIs They should be language and format neutral: http://my-dataset-name-language-and-format-neutral. Content negotiation can then be applied to get the desired language and format.

The table shows which multilanguage strategy applies to which property.

DCAT-AP CH scope

DCAT-AP CH uses only some of the classes of [[DCAT-AP]]. It also uses classes from [[FOAF]] for modeling the Agent class.

  • dcat:Catalog is a container of datasets in DCAT-AP CH.
  • dcat:Dataset is a container of distributions.
  • dcat:Distribution represents an accessible form of a dataset such as a downloadable file.
  • There are other classes in [[DCAT-AP]] that have not yet become part of DCAT-AP CH.

Vocabulary specification

This vocabulary specification has been added in order to provide a legend for the class property overview tables that have been added for each class.

The tables in the following sections include these columns:

  • Column Obl. ("Obligation"), denotes whether the class / property is mandatory ("M"), conditional (C) or optional (O). Conditional means that the property becomes mandatory under certain conditions.
  • Column MC specifies the maximum cardinality of the corresponding property (the minimum cardinality is always 1 for all mandatory properties, and 0 for the other ones). Properties having as range a plain literal can be specified multiple times, even though their max cardinality is 1, provided that each instance uses a different language tag. This case is marked by using an asterisk after the max cardinality value (e.g., "1*").
  • Column Conf. ("Conformance"), denotes whether the class / property definition is conformant ("Y") or not conformant (N) to [[DCAT-AP]].

Class: Catalog

Label Catalog
Obligation Mandatory
RDF Class dcat:Catalog
Definition A data catalog is a curated collection of metadata corresponding to the datasets that are hosted on a data portal.
Usage The catalog has a hierarchical structure and contains references to all the metadata for the data that can be accessed via the portal (these entries have the class dcat:Dataset). The datasets themselves refer to all distributions of the actual data (class dcat:Distribution). In DCAT-AP there is an additional class, dcat:CatalogRecord, which contains metadata on the catalog. This class has not been used so far in the implementation of DCAT-AP CH on opendata.swiss.
Conformance

The class definition conforms to [[DCAT-AP]], even though some of the properties defined for the class do not, as can be seen in the table below.

The implementation on opendata.swiss does not implement the catalog class with all mandatory properties and therefore does not conform either to DCAT-AP or to this specification (see ).

Class: Dataset


    

      
      

        Label
        Dataset
      

      

        Obligation
        Mandatory
      

      

        RDF Class
        dcat:Dataset
        
      

      

        Definition
        The metadata entry of this class describes a single thematically closed data collection,
          which is published by a responsible organization (person or institution). The dataset can be
          offered in different ways: as file in a text/csv format or as API Service. Each distribution
          SHOULD be described by a dcat:Distribution. In general, each dataset will have at least
          one distribution.
        
      


      

        Usage
        
          
The dataset will also list distributions if these are only available on an external website,
            so that the reference is set by the property dcat:landingPage of the dataset 

          
In exceptional cases, there can also be datasets that do not yet have distributions. This can be the case
            if the publication of these datasets is planned for a future date.

          
The property dcat:theme helps to group datasets by topics. Using a controlled vocabulary
            in the context of a data portal helps with the discoverability of the datasets by topic.
            Since in DCAT-AP the dcat:theme is optional, it can be ignored on export.

        		
      

      

        Conformance
        The definition of a dataset should be revised:
          to stay conformant to DCAT-AP the dcat:DataService class should be introduced.
          It must be considered that [[VOCAB-DCAT]] recommends that all distributions of one dataset
          should broadly contain the same data
          (see ).

Property: identifier


      

        
        

          Label
          identifier
        

        

          Obligation
          Mandatory
        

        

          Max card.
          1
        

        

          RDF Property
          dct:identifier
          
        

        

          Range
          rdfs:Literal
          
        

        

          Definition
          Unique identifier of the dataset.
            The identifier may be used as part of the URI of the dataset,
            but having it represented explicitly is still useful.
          
        

        

          Usage
          
It is necessary to uniquely mark each Dataset in order to be able to link it
            to similar datasets. The identifier of a dataset should therefore have
            the following structure:

            
ID_Datenset@ID_Datapublisher

            
Example: 
              124325@swisstopo
            

            
The identifier should be unique within Switzerland

            
The dataset can then be linked to other dataset by the element
              rdfs:seeAlso by adding the identifier to this property
              on other datasets.

            
Using rdfs:seeAlso for linking to other datasets makes it necessary that
              a data portal implementing this automatically takes care of consistency
              if datasets get changed or removed:

            
    
              
  • If the identifier of a dataset is changed: all seeAlso properties of
                other datasets on the portal should be changed accordingly
              
    • 
              
  • If a dataset is deleted: its identifier should be removed
                of the seeAlso of other datasets.
              
    • 
            

          		
        

        

          Conformance
          
The property is [[DCAT-AP]] conformant.

            
The exact recipe on how to form an identifier is closely related
              to the implementation on opendata.swiss and does not follow the best practice
              approach for DCAT identifiers. It is worth a
              discussion whether the definition should be generalized (see
              ).

Conformance to DCAT-AP


  

    
This document is the preparation for version 2 of DCAT-AP CH. The goal is to reach conformity with DCAT-AP
      while remaining in line with the Swiss demands of [[OGD]] and [[NaDB]]. 

    
Therefore this
    section explains the conformance criteria and points out conformance violations, so that they can
      be further discussed and hopefully solved by the community around DCAT-AP CH.

  


  

    
Conformance Criteria


    
DCAT-AP CH and DCAT-AP are both application profiles for DCAT.

    
The [[DCAT-PROFILE-GUIDANCE]] states that application profiles may form hierarchies.

    
For interoperabilty reasons DCAT-AP CH aims to be a subprofile of DCAT-AP. That way
    the data using DCAT-AP CH can be harvested by the European dataportal without loosing
    important information on its metadata.


    
The following diagramm captures the relationship between DCAT, DCAT-AP and DCAT-AP CH:

    

      classDiagram

      DCAT -->  DCAT_AP: is_profile_of
      DCAT_AP --> DCAT_AP_CH: is_subprofile_of
      DCAT --> DCAT_AP_CH: is_profile_of
      class DCAT {
        - defines classes
        - defines properties
        - base vocabulary
      }
      class  DCAT_AP {
        - mandatory/recommended classes
        - mandatory/recommended properties
        - mandatory/recommended vocabulary
      }
      class DCAT_AP_CH {
        - can restrict more than DCAT_AP
        - cannot restrict less than DCAT_AP
      }
    


    
DCAT-AP CH must conform to both DCAT and DCAT-AP.
      This conformance of DCAT-AP CH means in detail:

    
    
      
  • Conformance to DCAT: All classes and properties MUST be used in a way that they were meant for by the DCAT vocabulary:
      For example: DCAT vocabulary states that the distributions of a dataset should only differ in resolution,
      language and format but not in the content that they cover. Therefore an attrbitute such as
        dct:coverage on dcat:Distribution hurts that principle and is discusses
        in this section as being not conformant to DCAT. See
         for the details.
      
    • 
      
  • Conformance to DCAT:
        DCAT-AP CH MAY introduce Swiss specific properties and vocabulary, such as the cantons of Switzerland, that
        don't exist in the DCAT vocabulary.
        It should however not invent properties, that are alreay in DCAT expressed with different vocabulary. In that case the DCAT vocabulary
        should be used instead.
        For example the Swiss rdfs:seeAlso is used
        to express Relationships between Datasets. But DCAT already has vocabulary to express
        those relationships. Therefore these Swiss constructs are also considered to
        not conform to DCAT. They wont be understood on an international or European level.
        See .
      
    • 
      
  • Conformance to DCAT-AP:
        DCAT-AP CH can restrict more that DCAT-AP regarding mandatory fields and mandatory vocabulary, but it can't
        restrict less. For example currently DCAT-AP CH uses DCAT vocabulary for many properties, such as dct:language,
        dct:accrualPeriodicty. This usage conforms DCAT, but not DCAT-AP. Therefore these vocabulary usages are also
        considered to be a conformance issue. See  and
        .

Concept of a Dataset issue



      
Some problems that we currently face, for example the need of the attribute dct:coverage
        for dcat:Distribution come from the fact, that we we misinterpret what is meant by a dataset: in [[VOCAB-DCAT]]
        there is a note
        dcat:Distribution
      that states that the Distributions of a dataset should contain the same content, but in different
      format. If there is a series of different content it should rather be modeled as a datasets that relate
      to each other. 

      
It seems painful to change this defintion what a dataset is now, but in the long run we will be better
        of if we adapt to [[VOCAB-DCAT]]'s generel concepts. Only this will enable us to profit in the best
        way from its further development.
