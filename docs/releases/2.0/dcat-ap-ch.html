This document is the preparation for version 2 of DCAT-AP-CH. The goal is to reach conformity with the DCAT-AP while remaining in line with the Swiss demands of OGD and NaDB.

Currently the active version of DCAT-AP-CH is still version 1.0. This is just a draft to be completed.

DCAT is an RDF vocabulary designed to facilitate interoperability between data catalogs published on the Web. It enables a publisher to describe datasets and data services in a catalog using a standard model and vocabulary that facilitates the consumption and aggregation of metadata from multiple catalogs. This can increase the discoverability of datasets and data services. It also makes it possible to have a decentralized approach to publishing data catalogs and makes federated search for datasets across catalogs in multiple sites possible using the same query mechanism and structure.

DCAT-AP is the European application profile for DCAT that aims for interoperability within Europe. It specifies conventions on the usage of DCAT, that make sure metadata can be imported into the Europen Dataportal, in a way, where the metadata information is preserved and can be still understood by a visitor of that Dataportal.

DCAT-AP-CH is the Swiss application profile customizes DCAT for the needs of Swiss government agencies. One Swiss specialty are the 3 official languages: German, French and Italian, that are all supported by the DCAT-AP-CH

This document has as a first goal to make the DCAT-AP-CH version 1 into an HTML document, so that discussing it by the community and further developing it is facilitated. Once this transfer of the current standard is ready, the actual work of adapting it to the European DCAT-AP will start.

DCAT-AP-CH history

DCAT-AP-CH is the DCAT Applicationprofile for Dataportals in Switzerland. It serves to describe Data published by Swiss governemnt and other organizations. It was developped in parallel with the OGD portal of the Swiss Federal office of Statistics in 2016. It was last updated in 2018.

External terms

DCAT-AP-CH incorporates terms from pre-existing vocabularies where stable terms with appropriate meanings could be found, Changes to definitions in the references, if any, supersede the summaries given in this specification.

A data catalog conforms to DCAT-AP-CH if:

  • It contains all mandatory classes specified in the specification
  • An RDF description of the catalog itself is available
  • All classes and properties defined in the catalog are used in a way consistent with the semantics declared in this specification.
DCAT-AP-CH compliant catalogs MAY include additional non-DCAT-AP-CH metadata fields and additional RDF data in the catalog's RDF description.

That DCAT-AP-CH as a DCAT profile conforms to DCAT-AP means that it is a specification for a data catalog that adds additional constraints to DCAT-AP. A data catalog that conforms to the DCAT-AP-CH also conforms to DCAT-AP. Additional constraints in a profile MAY include:

  • Cardinality constraints, including a minimum set of required metadata fields
  • Sub-classes and sub-properties of the standard DCAT classes and properties
  • Classes and properties for additional metadata fields not covered in DCAT vocabulary specification
  • Controlled vocabularies or IRI sets as acceptable values for properties
  • Requirements for specific access mechanisms (RDF syntaxes, protocols) to the catalog's RDF description

Multilanguages, URIs and Content Negociation

Switzerland is a multilingual country. Therefore supporting multiple languages is very important in DCAT-AP-CH. See the below on how multiple languages are handled in the Swiss Application profile:

  • Multiligual Literals: Properties of Range rdfs:Literal support multiple languages out of the box and the Swiss Profile makes use of this. It also adds some additional restraints on what languages are expected.
  • Content Negotian: Properties of Range rdfs:Resource are supposed to be URIs. It is good practice to use URIs that are language independent ao that content negotiation can be used at these URI to provide the desired language to users that follow them.

Best practice for Resource URIs They should be language and format neutral: http://my-dataset-name-language-and-format-neutral Then content negotiation can be applied to get the desired language and format.

The tables shows for which properties which multilanguage strategy applies:

Label RDF property Range Multilanguage Support
Catalog title dct:title rdfs:Literal Language encoded string
Catalog description dct:description rdfs:Literal Language encoded string
Catalog homepage foaf:homepage foaf:Document Content Negotiation
Dataset title dct:title rdfs:Literal Language encoded string
Dataset description dct:description rdfs:Literal Language encoded string
Dataset keyword dcat:keyword rdfs:Literal Language encoded string
Dataset landing Page dcat:landingPage foaf:Document Content Negotiation

Vocabulary specification


  
The tables in the following sections include these columns:

  
    
    
  • Column Obl. ("Obligation"), denotes whether the class /
      property is mandatory ("M"), conditional (C) or optional (O)
      -
      where conditional means "mandatory if given conditions are met".
    
    • 
    
  • Column MC specifies the maximum cardinality of the
      corresponding property (the minimum cardinality is always 1 for all mandatory properties, and 0 for the other
      ones). Properties having as range a plain literal, can be specified multiple times, even though their max
      cardinality is 1, provided that each instance uses a different language tag. These case is marked by using an
      asterisk after the max cardinality value (e.g., "1*").
    
    • 
    
  • Column Conf. ("Conformance"), denotes whether the class /
      property definition is conformant ("Y") or
      not conformant (N) to [[DCAT-AP]]
    
    •
