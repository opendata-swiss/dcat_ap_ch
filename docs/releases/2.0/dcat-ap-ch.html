The DCAT Application Profile for data portals in Switzerland DCAT-AP CH is an application profile of the DCAT vocabulary [[VOCAB-DCAT]]. It serves to describe data that has been registered on Swiss data portals. Its target group are managers of open data portals in Switzerland and the providers of these data. DCAT-AP CH has been used by the Open Government Data Portal of the Swiss Federal Statistical Office since 2016.

It is important to be aware that data portals don't provide the data themselves but only metadata on the data. This serves the purpose of making the data discoverable.

The DCAT Application Profile for data portals in Switzerland has striven from the start to be compatible with the Application Profile for data portals in Europe [[DCAT-AP]]. Full conformance has not yet been reached, but the current specification takes responsibility in making the existing conformance problems visible, so that they can be resolved.

About

The DCAT Application Profile for data portals in Switzerland (DCAT-AP CH) is an application profile of the DCAT vocabulary ([[VOCAB-DCAT]]). It serves to describe data that are registered on Swiss data portals.

DCAT [[VOCAB-DCAT]] is a semantic definition to describe data by the means of an RDF vocabulary. It allows for a decentralized approach regarding the publication of data by enforcing interoperability via the use of a common language to describe the data.

Since DCAT is only a language, it can be applied in many contexts. An application profile applies DCAT to one specific context. In the case of DCAT-AP CH the context is that of data portals in Switzerland.

This context alone would allow for a very flexible usage of DCAT in DCAT-AP CH, but Switzerland is also part of Europe and therefore the interoperability with the data portal of the European Union has been a topic from the start. Europe has it's own application profile [[DCAT-AP]], that has established a role of a de facto standard within Europe. Swiss data is also harvested and represented by the official portal for European data. The EU puts a lot of power behind further developing their application profile DCAT-AP. Therefore staying in line with the DCAT-AP is not only a goal in itself, but means that DCAT-AP CH will stay embedded in a bigger community of stakeholders.

Metadata

Neither DCAT-AP CH nor DCAT-AP describe data, only metadata. Metadata is by definition secondary information on the data: when and by whom were they published, which usage conditions apply, how often are they updated, whom to contact about them and where and how can they be accessed. See [[METADATA-DEF]].

DCAT vocabulary building blocks

In DCAT the vocabulary consists of classes and properties.

  • Classes are things on the internet: Not all of them have URIs [W3C-URI], but it is recommended to provide a URI for them. They are complex things like a person, an organization, a data collection, a website or a downloadable data file.
  • Classes have properties: The properties are the attributes describing these things. Some properties occur on more than one class, a title for example is a common attribute. Other properties are very specialized such as a file format that only makes sense for a datafile.
  • Properties can be simple or complex: Some properties are classes. For example an organization can have a website. Or a dataset can have a data publisher. In general a class can be recognized by its spelling: A property name starts with a lower case letter such as dcat:dataset (), while a class starts with a capital letter such as dcat:Dataset ().
Classes used in DCAT-AP CH

Classes and properties are used to deliver the metadata in a structured way: The exact specification of these classes in the Swiss context is the topic of this specification as it is the topic of DCAT-AP to define the structure in the European context:

  • The catalog class: This is the top level class used to describe the data of one provider. DCAT-AP CH and DCAT-AP are all about how this top level class is further structured.
  • The distribution class: This is the bottom class that describes the physical data that can be accessed. The goal of both DCAT-AP CH and DCAT-AP is to make these distributions discoverable by embedding them into a structure that is universally understood.
  • The dataset class: In DCAT-AP CH there is only the dataset class in between catalog and distributions. Datasets are supposed to bundle similar distributions in units that can be described together. DCAT and DCAT-AP on the other hand allow for a more complex structure between catalogs and distribution by adding additional classes such as a Catalog Record or a Data Service class.

The three classes above are the classes that are further specified in DCAT-AP CH ( ).

Interoperability

DCAT-AP CH aims at making data discoverable on the national and international level by providing the necessary metadata in a form that can be understood by both humans and machines.

A precondition this for interoperability is that the DCAT language is used correctly. Therefore this specification aims at enabling data publishers to specify their data catalog in a way that conforms to both DCAT-AP CH and DCAT-AP. It also gives guidance to data portal managers on how the data catalogs should be received and processed in a DCAT-AP CH and DCAT-AP conformant way.

Provider requirements

A data catalog conforms to DCAT-AP CH if:

  • An RDF description of the catalog is available
  • All classes and properties defined in DCAT-AP CH are used in a way consistent with the semantics declared in this specification.
  • For any additional classes and properties the conformance rules of DCAT ( DCAT Conformance Criteria) and [[DCAT-AP]] (Section 6.1 "Provider requirements") apply.

Receiver requirements

An application (data portal) conforms to DCAT-AP CH if:

  • It is able to process RDF catalogs that conform to DCAT-AP CH.
  • Processing means that it must accept incoming data and transparently provide this data to applications and services.
  • Processing does not prescribe whether and how the metadata is stored internally, displayed or made searchable.
  • For any additional classes and properties, especially those coming from [[DCAT-AP]] the conformance rules of [[VOCAB-DCAT]] and [[DCAT-AP]] (Section 6.2 "Receiver requirements") apply: the application must be able to import and export classes and properties that are well-formed according to the semantics of DCAT-AP correctly and pass them through as they are when harvested by the European data portal.

Localization of DCAT-AP

    
DCAT-AP CH is a sub-profile of [[DCAT-AP]]. Its task is that of localizing DCAT-AP. This means
      it adds and overwrites the specification of DCAT-AP
      where Swiss data providers have different needs in describing their data, that cannot be met
      by DCAT-AP.


    
The overwrites fall into the categories listed below:


    
    
     
  • Adding custom properties:  DCAT-AP CH MAY add custom classes and properties, that are
       useful in the Swiss context. Examples for this are
       ,
        and .
       Adding new properties has however to conform to the regulations given in .
     
    • 
     
  • Overriding requirements:  DCAT-AP CH MAY also overwrite the requirements for certain
       properties and classes. For example if DCAT-AP marks a property as optional, DCAT-AP CH may
       set this property to mandatory. Examples for this are 
       and . When overwriting the requirements of a property
       DCAT-AP CH must make sure that if metadata conforms to these changed requirements, it will also conform to the original requirements
       of DCAT-AP. This means in practice that the requirements in DCAT-AP CH may become stricter, but not less strict. For example the title
       of a dataset that is already mandatory in DCAT-AP can't be set to optional in DCAT-AP CH.
    • 
     
  • Overriding ranges and usage notes: DCAT-AP CH MAY also overwrite the range of a property or
       how a property or class SHOULD
       be used. But again this has to be done in a way, that honors the restrictions given by DCAT-AP.
       For example, when DCAT-AP makes the use of a controlled vocabulary mandatory for a property, then DCAT-AP CH can't recommend a
       different vocabulary, unless this vocabulary is a subset of the DCAT-AP vocabulary.
       But it can specify that a property has to be provided in a specific format as it does
       for . Or it can demand certain languages for
       a string field as done in .

  • In theory:  That the class or property is already well defined in DCAT-AP: no further adjustments
        need to be made for the Swiss context. It can be used as it is.
    • 
       
  • In practice:  Most classes and properties with good support on Swiss open data portals
         have been added in this specification with some Swiss adjustments. Therefore

       in the way a data publisher might expect ().

  • Once DCAT-AP CH becomes fully DCAT-AP conformant:

Scope of DCAT-AP CH


    
DCAT-AP CH only specifies three classes of DCAT-AP:

    
    
      
  • 
        dcat:Catalog is a repository that contains a collection of datasets published by
        a data publisher.
      
    • 
      
  • 
        dcat:Dataset is a collection of data, published or curated
        by a single source, and available for access or download in one or more formats.
      
    • 
      
  • 
        dcat:Distribution represents an accessible distribution of a dataset
        such as a downloadable file or an access page where the data can be received
      
    • 
    


    
DCAT-AP CH has three requirement levels:

    
    
      
  • mandatory is an exact match of the DCAT-AP's requirement level mandatory and means the
        information MUST be given.
    • 
      
  • conditional is close to DCAT-AP's requirement level recommended. [[DCAT-AP]]
        defines "recommended" in its section 2 in the way that the information MUST be given if it is available. [[DCAT-AP-CH-1]] uses
        "conditional" instead of "recommended", but includes no general definition of this requirement level and only explains
        it on a case per case basis. See  for a
      discussion of this custom requirement level of DCAT-AP CH.
      
  • optional is an exact match of the DCAT-AP's requirement level optional and means
      the information MAY be provided, but there is no need to do so, even if the information is available.

  Class Property Tables

  
For each class a property table has been added, that includes all properties of the class, that
    are specified in DCAT-AP CH:

  
    
    
  • Column Obl. ("Obligation"), denotes whether the class /
      property is mandatory ("M"), conditional (C) or optional (O).
     See  for details on the requirement levels.
    
    • 
    
  • Column MC specifies the maximum cardinality of the
      corresponding property (the minimum cardinality is always 1 for all mandatory properties, and 0 for the other
      ones). Properties having as range a plain literal can be specified multiple times, even though their max
      cardinality is 1, provided that each instance uses a different language tag. This case is marked by using an
      asterisk after the max cardinality value (e.g., "1*").
    
    • 
    
  • Column Conf. ("Conformance"), denotes whether the class /
      property definition is conformant ("Y") or
      not conformant (N) to DCAT-AP. (This column
      just considers the conformance of the definition, not the conformance of any implementation.)

        Mandatory
      

      

        RDF Class
        dcat:Distribution
        
      

      

        Definition
        A metadata entry of this class
          describes a distribution of the data.
        
      

      

        Usage
        
The distribution can contain all the data of the dataset or it can contain just a certain
          part of the data. For example: it can contain all data
          about the population in Switzerland or it can contain just on year of this data, say
          the year 2018.
          It can also contain the data in an alternative data format, for example a graphical
          presentation of the data for the years 1990 up to 2010.

      

      

        Conformance
        
Does not conform to [[DCAT-AP]] as some of its properties do not conform to DCAT-AP.

          
Also the usage note does not conform to DCAT, since in DCAT distributions SHOULD only differ
          in format, resolution or language but not in regards to their content
            ()

Conformance to DCAT-AP


  

    
This document is the preparation for version 2 of DCAT-AP CH. The goal is to reach conformity with DCAT-AP
      while remaining in line with the Swiss demands.

    
Therefore, this
    section explains the conformance criteria and points out conformance violations, so that they can
      be further discussed and solved by the community around DCAT-AP CH.

  


  

    
Conformance Criteria


    
DCAT-AP CH and DCAT-AP are both application profiles for DCAT.

    
The [[DCAT-PROFILE-GUIDANCE]] states that application profiles may form hierarchies.

    
For interoperability reasons, DCAT-AP CH aims to be a sub-profile of DCAT-AP. That way
    the data using DCAT-AP CH can be harvested by the European data portal without losing
    important information on its metadata.


    
The following diagram captures the relationship between DCAT, DCAT-AP and DCAT-AP CH:

    

      classDiagram

      DCAT -->  DCAT_AP: is_profile_of
      DCAT_AP --> DCAT_AP_CH: is_subprofile_of
      DCAT --> DCAT_AP_CH: is_profile_of
      class DCAT {
        - defines classes
        - defines properties
        - base vocabulary
      }
      class  DCAT_AP {
        - mandatory/recommended classes
        - mandatory/recommended properties
        - mandatory/recommended vocabulary
      }
      class DCAT_AP_CH {
        - can restrict more than DCAT_AP
        - cannot restrict less than DCAT_AP
      }
    


    
DCAT-AP CH must conform to both DCAT and DCAT-AP.
      This conformance of DCAT-AP CH means in detail:

    
    
      
  • Conformance to DCAT: All classes and properties MUST be used in a way that they were meant for by the DCAT vocabulary:
      For example: DCAT vocabulary states that the distributions of a dataset should only differ in resolution,
      language and format but not in the content that they cover. Therefore an attribute such as
        dct:coverage on dcat:Distribution hurts that principle and is discussed
        in this section as being not conformant to DCAT. See
         for the details.
      
    • 
      
  • Conformance to DCAT:
        DCAT-AP CH MAY introduce Swiss specific properties and vocabulary, such as the cantons of Switzerland, that
        don't exist in the DCAT vocabulary.
        It should however not introduce its own (custom) properties that are already expressed in DCAT with different vocabularies.
        In those cases the DCAT vocabulary has to be used.
        For example the Swiss rdfs:seeAlso is used
        to express relationships between datasets. But DCAT already has vocabulary to express
        those relationships. Therefore these Swiss constructs are not conforming to DCAT.
        These relationships cannot be interpreted on an international or European level.
        See .
      
    • 
      
  • Conformance to DCAT-AP:
        DCAT-AP CH can restrict more than DCAT-AP regarding mandatory fields and mandatory vocabulary, but it is not allowed
        to restrict less. For example currently DCAT-AP CH uses DCAT vocabulary for many properties, such as dct:language,
        dct:accrualPeriodicity. This usage conforms to DCAT, but not DCAT-AP. Therefore these vocabulary usages are also
        considered to be a conformance issue. See  and
        .

Learnings and Next Steps

    

      This section reflects solely the opinion of the editor. It is added to prepare the
      discussion about the next steps for DCAT-AP CH.
    

    

    
Learnings

        
    
          
  • Get and stay in touch with DCAT-AP community: The best impulses for this work were received
        in discussions with the DCAT-AP community. They bring the fresh view of being native to DCAT-AP, but strangers to
        DCAT-AP CH. Therefore many problems only became clearer in discussions with them. Good touchpoints
        are [[EUROVOC]] and [[DCAT-AP-GITHUB]].
    • 
      
  • The most important task of an application profile is localization:
        DCAT-AP CH is a sub-profile of DCAT-AP. So why is it necessary? DCAT-AP targets the European community, but
        many of its rules cannot be directly applied to the situation in Switzerland.
        For example, DCAT-AP has a strong focus on demanding the use of controlled
        vocabulary for many properties, but the European vocabulary often can't be used for Swiss metadata without further
        adaptions.
    • 
     
  • Don't add general custom vocabulary:
           This does not conform to DCAT's rules for vocabulary extension
          (DCAT:Conformance),
          but has been done by DCAT-AP CH. See 
          or .
          Interesting in that regard is that adding, for example, the property dct:coverage to
          dcat:Distribution has led to a different understanding of what kind of distributions a dataset can hold,
          see . These side effects
          will be hard to correct, but correction will be needed in order to profit
          from future developments in DCAT.
    • 
          
  • Without examples properties are very likely to be misunderstood:
          Both DCAT-AP CH and DCAT-AP currently lack examples.
             lists many issues where
            properties have been misunderstood. Examples are  and
            .
    • 
       
  • Don't mix in application logic:
         In DCAT-AP CH some properties contain application logic. See 
       and . This hurts interoperability, since
         the application logic is not known on the European or international level. It also stands in the
       way of the reuse of DCAT-AP CH for other applications.
    • 
         
  • The power of URIs is not used:
         URIs have a lot of potential in regards to identifying and localizing data. This potential
         is not yet used in DCAT-AP CH. For example, localization of a landing page should be done with content negotiation
           by using a language-neutral URI ().
         Relations between datasets should be expressed with DCAT vocabulary by connecting datasets by their URIs,
           as described in 
             DCAT:Relationships between datasets and other resources, instead of using application-bound identifiers for that purpose
           ().
         
    • 
        
  • The relationship to DCAT-AP needs to be explicitly defined in the specification: Providers of data and portal
          managers need guidance on how to understand what it means that a class or property is specified in DCAT-AP
          but not in DCAT-AP CH: Do providers of data need to look at both specifications in order to
          arrive at a well-formed data catalog? That would be the case if DCAT-AP CH does not include definitions for all
          mandatory and recommended classes and fields. DCAT-AP conformance is in many ways decided in the treatment
          of the gap between DCAT-AP CH and DCAT-AP ().
